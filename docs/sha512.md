# Documentation — SHA-512

---

## Vue d'ensemble

SHA-512 est un algorithme de hachage de la famille SHA-2, défini par la
RFC 6234 et la FIPS 180-4. Il produit un digest de 512 bits (128 chars hex).

---

## Architecture — comparaison avec SHA-256

| Paramètre | SHA-256 | SHA-512 |
|-----------|---------|---------|
| Taille bloc | 64 octets (512 bits) | 128 octets (1024 bits) |
| Taille mots | 32 bits (uint32_t) | 64 bits (uint64_t) |
| Registres | 8 × uint32_t | 8 × uint64_t |
| Schedule | W[64] | W[80] |
| Rounds | 64 | 80 |
| Constantes K | 64 × uint32_t | 80 × uint64_t |
| Padding jusqu'à | 56 octets | 112 octets |
| Taille encodée | 8 octets | 16 octets |
| Hash final | 64 chars hex | 128 chars hex |
| Endianness | big-endian | big-endian |

---

## Pipeline

```
input
    ↓
sha512_init()      → H0..H7 initialisés (uint64_t)
    ↓
sha512_process()   → blocs de 128 octets → sha512_transform()
    ↓
sha512_padding()   → 0x80 + zéros + 16 octets taille big-endian
    ↓
sha512()           → 8 × word_to_hex64 → 128 chars
```

## Valeurs initiales — `sha512_init`

Les 8 registres sont les **64 premiers bits** de la partie fractionnaire
des racines carrées des 8 premiers nombres premiers.

```
SHA-256 utilise 32 bits de frac(√p)
SHA-512 utilise 64 bits de frac(√p) — plus de précision

√2  → H0 = 0x6a09e667f3bcc908  (SHA-256 H0 = 0x6a09e667)
√3  → H1 = 0xbb67ae8584caa73b
√5  → H2 = 0x3c6ef372fe94f82b
√7  → H3 = 0xa54ff53a5f1d36f1
√11 → H4 = 0x510e527fade682d1
√13 → H5 = 0x9b05688c2b3e6c1f
√17 → H6 = 0x1f83d9abfb41bd6b
√19 → H7 = 0x5be0cd19137e2179
```

## Message schedule — `build_schedule`

SHA-512 étend 16 mots de 64 bits en **80 mots** (SHA-256 : 16→64) :

```c
W[i] = SHA512_SWAP(M[i])                              // i = 0..15
W[i] = σ1(W[i-2]) + W[i-7] + σ0(W[i-15]) + W[i-16]  // i = 16..79
```

### Macros sigma — rotations différentes de SHA-256

```c
// SHA-256 sigma schedule
SHA256_s0(x) = ROTR32(x,7)  ^ ROTR32(x,18) ^ (x >> 3)
SHA256_s1(x) = ROTR32(x,17) ^ ROTR32(x,19) ^ (x >> 10)

// SHA-512 sigma schedule — rotations plus grandes sur 64 bits
SHA512_s0(x) = ROTR64(x,1)  ^ ROTR64(x,8)  ^ (x >> 7)
SHA512_s1(x) = ROTR64(x,19) ^ ROTR64(x,61) ^ (x >> 6)
```

```c
// SHA-256 Sigma rounds
SHA256_S0(x) = ROTR32(x,2)  ^ ROTR32(x,13) ^ ROTR32(x,22)
SHA256_S1(x) = ROTR32(x,6)  ^ ROTR32(x,11) ^ ROTR32(x,25)

// SHA-512 Sigma rounds
SHA512_S0(x) = ROTR64(x,28) ^ ROTR64(x,34) ^ ROTR64(x,39)
SHA512_S1(x) = ROTR64(x,14) ^ ROTR64(x,18) ^ ROTR64(x,41)
```

## Constantes K[80]

80 constantes uint64_t — racines cubiques des 80 premiers premiers :

```c
K[0]  = 0x428a2f98d728ae22ULL  // ∛2
K[1]  = 0x7137449123ef65cdULL  // ∛3
// ...
K[79] = 0x6c44198c4a475817ULL  // ∛397
```

SHA-256 utilise 64 constantes uint32_t — SHA-512 en utilise 80 en uint64_t.

## `t_sha512_regs` — union des registres

Identique à `t_sha256_regs` mais en uint64_t :

```c
typedef union   u_sha512_regs
{
    struct { uint64_t a, b, c, d, e, f, g, h; };
    uint64_t    r[8];
}               t_sha512_regs;
```

## `sha512_process` — différences avec SHA-256

```c
// SHA-256
while (len >= 64)   { ... sha256_transform ... }
fill = 64 - ctx->used;

// SHA-512
while (len >= 128)  { ... sha512_transform ... }
fill = 128 - ctx->used;
```

Seule la taille du bloc change — la logique est identique.

---

## `sha512_padding` — différences avec SHA-256

```
SHA-256 padding :
    0x80 → zéros jusqu'à used == 56 → 8 octets taille

SHA-512 padding :
    0x80 → zéros jusqu'à used == 112 → 16 octets taille
    (8 octets de zéros pour les bits hauts + 8 octets de taille réelle)
```

```c
// SHA-256
while (ctx->used != 56)
    sha256_process(ctx, "\0", 1);
i = 7; while (i >= 0) { byte encodé sur 8 octets... }

// SHA-512
while (ctx->used != 112)
    sha512_process(ctx, "\0", 1);
i = 7; while (i >= 0) { byte = 0x00; ... }  // 8 octets hauts = 0
i = 7; while (i >= 0) { byte = taille... }   // 8 octets bas = taille
```

### Cas limite — message de 112 octets

```
SHA-256 cas limite : 56 octets (0x80 → exactly 56 → 2ème bloc)
SHA-512 cas limite : 112 octets (0x80 → exactly 112 → 2ème bloc)

après 0x80 :
    used == 112 → boucle while ignorée
    16 octets de taille → used = 128 → transform → used = 0
```

## `sha512` — hash final

```c
// SHA-256 : 4 chars par nibble × 8 mots = 64 chars
word_to_hex(uint32_t word, char *buf)   // buf[8]

// SHA-512 : 4 chars par nibble × 16 mots 4-bit = 16 chars par mot
word_to_hex64(uint64_t word, char *buf) // buf[16]

// SHA-256 : 8 mots × 8 chars = 64 chars total
// SHA-512 : 8 mots × 16 chars = 128 chars total
```

---

## `SHA512_SWAP` — swap 8 octets

```c
// SHA-256 swap 4 octets
SHA256_SWAP(x) = (x>>24) | ((x>>8)&0xFF00) | ((x<<8)&0xFF0000) | (x<<24)

// SHA-512 swap 8 octets — même principe sur 64 bits
SHA512_SWAP(x) = (x>>56) | ((x>>40)&0xFF00ULL) | ... | (x<<56)
```

## Ce qui est réutilisé de SHA-256

```
Architecture dispatch    ✓  une ligne dans hashers[]
run_hash / display_hash  ✓  rien à changer
t_sha256_regs pattern    ✓  copié en uint64_t → t_sha512_regs
logique process          ✓  64 → 128
logique padding          ✓  56/8 → 112/16
logique final            ✓  word_to_hex32 → word_to_hex64
fonctions CH / MAJ       ✓  identiques sur uint64_t
```
